# 给 Agent 的说明（项目负责人 / 代码小白用，纯文本简化版）

用法：每次和新 Agent 对话时，可以在第一条消息里提到本文件（例如说「按 AGENT_小白对接说明.md 里的规则和我协作」），或者复制下面的开场白。

一、开场白（复制到新对话第一条消息）：
我是这个项目的负责人，代码不太熟。请按下面方式和我协作：
1. 用大白话解释，少用术语；必须用术语时顺带一句人话解释。
2. 动手改代码前，先简短说你要做哪几件事、动哪些文件，等我确认或说「继续」再改。
3. 遵守项目里的规矩：README、TASKS.md、CONTRACTS.md、STATUS.md。不确定时先问我或先读这些文件。
4. 一次只做一件事或一个小目标，做完用一两句话总结「做了啥、你要怎么验收」。
我这次想让你做的是：[在这里写你的具体需求]。

二、你可以怎么描述需求（举例，按本项目：个人博客）：
- 如果你想要「博客列表分页」，可以说：首页或博客列表先只显示一部分，下面有个按钮，点一下再加载更多。
- 如果你想要「新文章从文件读」，可以说：增加一篇文章，正文从 content/posts/xxx/body.md 读，和现有做网站教程那篇一样。
- 如果你想要「文档让人能照着做」，可以说：在 README 里写清楚本地怎么跑、怎么加文章、怎么部署。
- 如果你想要「接口格式统一」，可以说：若以后加 API，成功就返回 ok 和 data，失败就返回 error 里带 code 和 message。

三、三条「不用懂代码」的用法：
1. 用文件指路，让 Agent 自己读。你可以说：请先看 TASKS.md 和 meta/CHECK_REPORT.md，然后告诉我现在该先做哪几件事、有没有风险。Agent 会自己读文件，再用人话总结给你。
2. 用验收驱动，不盯实现。你可以说：某任务做完了，请你按 TASKS 里的验收条件帮我检查一遍，用大白话告诉我通过没有、哪里还有问题。你只要看结论，不用懂代码。
3. 固定一个“汇报格式”。每次让 Agent 做完事后，用这样的格式结束：做了啥（用一两句话），你该怎么验收（在浏览器点哪里、看到什么算对），有没有要注意的（例如要重启或清缓存）。这样你照着做就能确认。

四、和多个 Agent 协作时的简单原则：
- 可以一个任务配一个对话，例如 T001 给前端对话，T002 给后端对话。
- 在新对话里说：我要你做 T00X（任务名），规矩在 TASKS.md，只许改该任务允许的文件，做完在 STATUS.md 里按模板补一条。
- 不同 Agent 各干各的，你只负责指明任务、看验收结果、决定合并不合并。
