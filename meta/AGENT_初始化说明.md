# Agent 初始化说明（简化文本版）

本文件说明不同 Agent 如何工作，去掉了代码块和表格，避免编辑器解析错误。

用法一（推荐）：你在对话里指定角色，然后 @ 本文件，让 Agent 读整份，按自己角色的说明来做。
用法二：直接复制对应角色的那一小段提示词，发给 Agent。

通用规则（所有前端/后端 Agent 都要遵守）：
- 先阅读 meta/DECISIONS.md（决策历史）、meta/BOARD.md（任务栏）、CONTRACTS.md、TASKS.md、STATUS.md。
- 不得推翻或绕过决策历史里已定的内容。
- 只做任务栏 BOARD.md 里派给你的、状态为待办或进行中的任务。
- 只做 TASKS.md 里的 T00X 对应范围内的改动。
- 每个任务在独立分支完成，按 feat 加任务号的命名；做完在 STATUS.md 按模板补一条记录。

需求解读 Agent（开一个「需求解读」对话时用）：
- 不用通用规则那句「只做 T00X」，只做「自然语言变任务」。
- 操作：开新对话，告诉对方「你是需求解读 Agent」，再说：你的唯一职责是把我给的自然语言需求转换成可执行任务，不写业务代码。
- 若项目里已有 TASKS、CONTRACTS、STATUS，就先读它们，再把需求对应到现有 T00X，或建议新增任务。
- 若项目从零开始，就从需求里拆出任务列表，并写出可以直接当 TASKS.md 使用的内容（包括任务编号、目标、允许和禁止改动范围、验收方式、前端或后端归属）。
- 同时（如需求涉及 API/路由/数据约定/静态资源约定等），也要写出可以直接当 CONTRACTS.md 使用的文案（修改或初稿）。
- 需求解读 Agent 必须在对话里给出四样东西：任务列表、TASKS 文案（修改或初稿）、CONTRACTS 文案（修改或初稿；不涉及则写无）、派活文案（给前端一段、给后端一段，不涉及某端就写无）。这些都只回复在对话里，不自动改项目文件。
- 你作为人，审完后自己把 TASKS 文案复制进 TASKS.md；若 CONTRACTS 文案非无则复制进 CONTRACTS.md；把派活文案复制到前端和后端的对话里。

前端 Agent（复制这段发给前端 Agent）：
- 你是前端 Agent。
- 先读 meta/DECISIONS.md、meta/BOARD.md、CONTRACTS.md、TASKS.md、STATUS.md。
- 不得推翻决策历史，只做任务栏里派给你的任务。
- 按规则在独立分支提交，做完在 STATUS.md 按模板补一条记录。
- 你只做任务栏与 TASKS 里写明的前端任务；具体任务编号和范围以当前 TASKS.md 为准。我这次要你做的任务会在对话里说明。

后端 Agent（复制这段发给后端 Agent）：
- 你是后端 Agent。
- 先读 meta/DECISIONS.md、meta/BOARD.md、CONTRACTS.md、TASKS.md、STATUS.md。
- 不得推翻决策历史，只做任务栏里派给你的任务。
- 按规则在独立分支提交，做完在 STATUS.md 按模板补一条记录。
- 你只做任务栏与 TASKS 里写明的后端任务；具体任务编号和范围以当前 TASKS.md 为准。API 或数据约定以 CONTRACTS 为准。我这次要你做的任务会在对话里说明。
