# 使用路线与关键梳理（纯文本简化版）

一、各文件的关键（一句话说明，谁读谁更新）：
- CONTRACTS.md：API 和数据契约，成功时 ok 和 data，失败时 ok 和 error，服务端补齐敏感字段，密钥不进前端；所有 Agent 动手前必读，由需求解读 Agent（AI）起草/更新文案，人只需审阅确认后将其纳入规则（写入文件）。
- TASKS.md：任务定义，包含 T00X 编号、目标、允许和禁止改动范围、验收方式，一任务一分支；所有 Agent 必读，由人或需求解读产出后人贴入。
- STATUS.md：任务完成后的记录本，每个任务完成都按模板补一条，写分支、改动、验收、风险、回滚；所有 Agent 必读，由前端或后端在做完任务后补。
- meta/DECISIONS.md：决策历史，记录已定过的事，Agent 不得推翻，需求审完不管过不过都记一条；所有 Agent 动手前必读，由人维护。
- meta/BOARD.md：任务栏，当前可做任务清单，只有人审通过的任务才加进当前栏，做完或移出记在历史里；所有 Agent 必读，由人维护。
- meta/AGENT_初始化说明.md：初始化规则，包含需求解读、前端、后端三个角色的提示词；由人让 Agent 读或复制对应段落。
- meta/AGENT_小白对接说明.md：给代码小白用的协作说明，包含开场白、怎么提需求、怎么验收；由人使用或在新对话里 @ 出来。
- meta/CHECK_REPORT.md：检查报告文件，对照 CONTRACTS、TASKS、STATUS 检查实现情况并给出验收建议；人或检查 Agent 使用。

约束两句：决策历史文件说「不能改什么」，任务栏文件说「只能做哪些」。前端和后端 Agent 只读这两个文件，不修改它们。

二、一条完整的使用路线：

路线 A：有原始需求（客户或产品给的一段自然语言）。
1. 需求解读：开一个需求解读对话，按 meta/AGENT_初始化说明里对需求解读 Agent 的说明，让它把自然语言拆成任务列表、TASKS 文案、CONTRACTS 文案和派活文案。
2. 人审加记决策：你看一遍产出是否合理，在 meta/DECISIONS.md 里按格式记一条，写清是审通过还是审不通过以及结论，不管过不过都要记。
3. 审通过时更新任务栏和文件：把通过的任务写入 meta/BOARD.md 的当前任务栏（写负责人和状态），如果有新的 TASKS 文案，就复制到 TASKS.md 里；如果有新的 CONTRACTS 文案，就复制到 CONTRACTS.md 里。
4. 派活：把需求解读产出的前端派活文案复制到前端 Agent 对话，把后端派活文案复制到后端 Agent 对话。
5. 前端和后端执行：各自先读 meta/DECISIONS、meta/BOARD、CONTRACTS、TASKS、STATUS，只做任务栏里派给自己的任务，在独立分支上开发，做完在 STATUS.md 按模板补记录，必要时更新 meta/BOARD.md 的状态或历史。
6. 人验收：按 STATUS 或 TASKS 里的验收步骤检查，通过就合并分支，不通过就退回给对应 Agent 修改。

路线 B：任务已经在 TASKS 和 BOARD 里定好了。
1. 人直接派活：开前端和后端对话，用 meta/AGENT_初始化说明里的前端或后端说明，告诉它们读相应文件，并说清这次要做哪一个 T00X。
2. 执行：前端和后端同路线 A 的第 5 步。
3. 验收：同路线 A 的第 6 步。

路线 C：从零开始（项目里还没有 TASKS、CONTRACTS、STATUS）。
1. 需求解读：同路线 A 的第 1 步，但会产出一份完整的 TASKS 初稿、CONTRACTS 初稿和对应的任务列表。
2. 人审加记决策：同路线 A 的第 2 步。
3. 人建文件：把 TASKS 初稿复制到项目里当 TASKS.md，把 CONTRACTS 初稿复制到项目里当 CONTRACTS.md（如有需要），并根据通过的任务填好 meta/BOARD.md。
4. 后续派活和执行：再按路线 A 的第 4、5、6 步进行。

三、一句话记住这套流程：
有新需求时，先用需求解读 Agent 把自然语言变成任务和 TASKS，再由你审和记到决策历史，通过的写进任务栏，然后把派活文案发给前端和后端；前端和后端只读规则文件，只做栏里的任务，做完在 STATUS 里记一条，最后由你验收和决定合并。
